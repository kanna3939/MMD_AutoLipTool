# MS13 Block 分解文書
## wxPython 最小基盤移行
## Antigravity 向け実装分解文書

## 0. 文書目的

本書は、`MS13: wxPython 最小基盤移行` を  
**Antigravity が実装単位へ分解しやすい粒度で block 化するための文書**である。

本書の役割は次のとおり。

- MS13 の責務境界を固定する
- MS13 を block 単位へ分解する
- block ごとの対象・非対象・完了条件を明示する
- MS14 以降へ持ち越す内容を先に切り分ける
- 実装中に論点が拡散しないよう、MS13 の到達点を固定する

本書は詳細コード実装書ではない。  
本書は、**MS13 の実装分割と進行順を固定するための block 設計文書**として扱う。

---

## 1. 本文書で固定する前提

本書は、以下の判断結果を前提として作成する。

### 1.1 GUI 移行方式
- 既存 Qt GUI は置き換え前提とする
- wx GUI を主系として順次差し替える
- Qt GUI は MS13 開始時点で凍結扱いとする

### 1.2 MS13 の到達点
MS13 の完了条件は、以下まで含める。

- wx 版アプリが起動する
- メインフレームが成立する
- 最低限のメニューが成立する
- 基本ボタン配置が成立する
- 設定ロードが wx 版起動導線へ接続される

ただし、以下は MS13 に含めない。

- TEXT/WAV 読込ダイアログ導線
- recent files
- VMD 保存導線
- 波形 / Preview 表示
- 再生同期
- オートスクロール
- 実運用 worker 処理
- スプラッシュ画面

### 1.3 settings 方針
- 既存 ini / key 構造は可能な限り維持する
- settings schema の再設計は MS13 では行わない
- GUI 移植と schema 変更を同時に進めない

### 1.4 スプラッシュ方針
- スプラッシュは MS13 非対象とする
- 起動骨格の成立を最優先し、演出要素は後段へ送る

### 1.5 worker 方針
- MS13 では実運用 worker を完成させない
- ただし、将来接続用の最小受け皿は用意してよい
- 本格的な非同期処理導線は MS14 以降で扱う

---

## 2. MS13 の位置づけ

MS13 は Ver0.4 系の最初のマイルストーンであり、  
**Qt / PySide6 ベースの GUI から wxPython ベースの GUI へ移行する最初の土台作業**である。

MS13 の本質は、機能追加ではない。  
MS13 の本質は、**今後の wx 版実装を進めるための最小起動基盤を成立させること**である。

このため、MS13 は次の性質を持つ。

- 完成した製品機能を増やす段ではない
- 既存 GUI の全面 parity を目指す段ではない
- 後段機能が乗る「骨格」を安定させる段である

---

## 3. MS13 の責務境界

## 3.1 MS13 で行うこと

- wx アプリ起動骨格を作る
- wx メインフレームを作る
- 現行 main window に対応する最小レイアウトを置く
- 基本メニューを置く
- 基本ボタン列を置く
- ステータス表示の最小導線を置く
- 既存 settings load を新 GUI 起動導線へ接続する
- 今後の非同期処理導入に備えた最小接続点を設ける

## 3.2 MS13 で行わないこと

- Qt GUI の改善
- Qt / wx の二重保守
- settings schema 再設計
- TEXT/WAV 読込ダイアログ導線の完成
- recent files
- 保存導線
- 波形表示
- Preview 表示
- 再生同期
- オートスクロール
- 実 worker 処理の完成
- スプラッシュ
- mp3 対応
- 音声のみ入力
- 品質改善
- 英語対応

---

## 4. MS13 の到達イメージ

MS13 完了時のイメージは以下のとおり。

- `wx.App` ベースでアプリが起動する
- メインフレームが表示される
- メニューが表示される
- 操作ボタン列が表示される
- ステータス表示が存在する
- 起動時 settings がロードされる
- settings のうち MS13 時点で必要な見た目状態が起動時に適用される
- 後続マイルストーンで機能を差し込める構造ができている

ただし、MS13 完了時点では「まだ使い切れるツール」にはしない。  
MS13 は、**使える製品への復帰ではなく、wx 主系の骨格成立**を完了条件とする。

---

## 5. Block 全体像

MS13 は、以下の block 構成で進める。

- `MS13-B1`: 移行前提固定と wx 起動入口作成
- `MS13-B2`: メインフレーム最小骨格作成
- `MS13-B3`: メニュー / 基本操作列 / ステータスの最小配置
- `MS13-B4`: settings load 接続と起動時適用
- `MS13-B5`: 将来拡張の接続点整理
- `MS13-B6`: MS13 統合整理と完了確認

依存関係は以下の順を基本とする。

`B1 -> B2 -> B3 -> B4 -> B5 -> B6`

必要があっても、B4 より前に schema をいじらない。  
必要があっても、B5 より前に worker 本体へ広げない。

---

## 6. Block 詳細

## MS13-B1
## 移行前提固定と wx 起動入口作成

### 目的
Qt 主系を凍結し、wx 主系へ移るための最小起動入口を作る。

### 主対象
- `src/main.py`
- wx 起動用の新規 app 入口
- GUI 初期化の最小導線

### この block でやること
- wxPython を主系とする起動入口を作る
- `wx.App` ベースの起動導線を作る
- 既存 Qt 起動経路との責務分離を明確にする
- Qt 側を今後の改修対象から外す前提を文書と実装上で揃える

### この block でやらないこと
- メイン画面の完成
- メニュー構築
- 設定適用
- 実機能導線
- スプラッシュ

### 完了条件
- wx 版アプリが起動する
- 起動入口が wx 主系として成立する
- Qt 主系と混線しない構造が作られている

### Antigravity 向け補足
- ここでは「起動すること」が主目的
- 見た目の完成度は求めない
- 既存 `main.py` の責務をそのままコピーするのではなく、wx 版に必要な最小責務へ絞る

---

## MS13-B2
## メインフレーム最小骨格作成

### 目的
現行 MainWindow に相当する wx メインフレームの最小骨格を成立させる。

### 主対象
- wx メインフレーム
- ルートレイアウト
- 最低限の panel / container 骨格

### この block でやること
- メインフレームを定義する
- 今後の UI 部品が載る大枠レイアウトを置く
- 左右 / 上下の大枠構造を表現できる土台を作る
- status 領域、操作領域、中央表示領域の置き場所を確保する

### この block でやらないこと
- 波形埋め込み
- Preview 埋め込み
- 実際の file dialog
- 実際の処理接続

### 完了条件
- メインフレームが表示される
- 今後の UI 配置先が確保される
- ルートレイアウトが極端な暫定実装になっていない

### Antigravity 向け補足
- 後続 block の配置替えが少なくて済むよう、root layout は早い段階で整える
- この段では、機能部品の完成よりも「置き場の安定化」を重視する

---

## MS13-B3
## メニュー / 基本操作列 / ステータスの最小配置

### 目的
起動しただけで終わらないよう、最小の操作導線を見える形にする。

### 主対象
- メニューバー
- 基本操作ボタン列
- ステータス表示

### この block でやること
- 現行 GUI に対応する最小メニュー構造を置く
- 基本ボタン列を置く
- ステータス表示領域を置く
- まだ実装しない操作には placeholder 導線を与える
- メニュー / ボタン / ステータスの責務境界を明確にする

### この block でやらないこと
- ファイル読込の実処理
- recent files
- 保存
- 再生
- 波形 / Preview
- 実 worker 接続

### 完了条件
- メニューが表示される
- 基本操作ボタン列が表示される
- ステータス表示ができる
- 後続 block でイベント接続しやすい構造になっている

### Antigravity 向け補足
- 実際に動かないボタンがあってもよいが、将来の接続先が曖昧にならないよう命名と責務を揃える
- この段では parity 完成を目指さない

---

## MS13-B4
## settings load 接続と起動時適用

### 目的
既存 settings schema を維持したまま、wx 起動導線へ settings load を接続する。

### 主対象
- `src/gui/settings_store.py`
- wx 版起動時の settings load
- 起動時に適用すべき最小 UI 状態

### この block でやること
- 既存 settings load を流用する
- 既存 ini / key 構造を維持する
- 起動時に必要な最低限の状態を反映する
- schema 再設計を行わずに接続する

### この block でやることの例
- ウィンドウサイズ
- language の初期決定
- theme の起動時反映の検討
- 将来の settings save へ繋がる土台確認

### この block でやらないこと
- settings schema 改変
- recent files の full 復元
- splitter 比率の精密再現
- 現行設定項目すべての完全反映

### 完了条件
- settings load が wx 起動導線に接続される
- 既存 settings を壊さず起動できる
- 起動時の最低限の UI 状態が反映される

### Antigravity 向け補足
- schema を触りたくなっても MS13 では避ける
- 「今必要な起動時反映」に絞る
- save 側の完成は MS14 以降へ送ってよい

---

## MS13-B5
## 将来拡張の接続点整理

### 目的
MS14 以降で必要になる処理導線を差し込みやすくするため、最小の接続点だけ用意する。

### 主対象
- worker 受け皿
- イベント接続点
- UI 更新呼び出し口
- 将来の view / preview / playback の差し込み余地

### この block でやること
- 実 worker ではなく、将来接続用の最小受け皿を決める
- UI 更新の入口を一箇所化しやすい構造を意識する
- file open / process / save / playback のイベント受け口を整理する
- MS14 以降で処理本体を差し込みやすい構造にする

### この block でやらないこと
- 実際の非同期処理完成
- 波形描画
- Preview 描画
- playback 実装
- オートスクロール

### 完了条件
- 後続マイルストーンで接続可能な最小受け皿が存在する
- 今後の block 追加で root 設計を崩しにくい
- 実 worker 未完成でも構造的な見通しが立つ

### Antigravity 向け補足
- ここで作るのは「将来の入口」であって「今の完成品」ではない
- 実処理まで書き始めると MS13 の責務を超える

---

## MS13-B6
## MS13 統合整理と完了確認

### 目的
MS13 の各 block を統合し、MS14 へ受け渡せる状態へ整理する。

### 主対象
- block 間整合
- 命名整理
- フォルダ / モジュール配置整理
- MS14 への引継ぎ情報整理

### この block でやること
- block 間の重複や暫定実装を整理する
- 明らかな naming ぶれを直す
- 今後の GUI 拡張で使う基本部位を再確認する
- MS14 へ渡す「現時点でできること / まだできないこと」を明記する

### この block でやらないこと
- 機能追加
- parity 回復
- file dialog 実装
- recent files 実装
- worker 実装本体

### 完了条件
- MS13 各 block が一通り統合されている
- MS13 の責務外を無理に抱え込んでいない
- MS14 を開始できる基盤として整理されている

### Antigravity 向け補足
- 最後に scope creep を削る block として扱う
- ここで機能を足さず、境界を綺麗にすることを優先する

---

## 7. Block 間の依存関係

## 7.1 基本順序
- B1 が起点
- B2 は B1 依存
- B3 は B2 依存
- B4 は B1 / B2 / B3 の最低限成立後
- B5 は B3 / B4 の後
- B6 は全 block の統合段

## 7.2 順序を崩しにくい理由
- B1 より前に layout を作る意味が薄い
- B2 がないと B3 の配置先が安定しない
- B3 がないと B4 の起動時反映対象が曖昧になる
- B4 の前に settings 再設計へ寄ると scope が崩れる
- B5 の前に実 worker へ進むと MS14 と混ざる

---

## 8. MS13 完了条件

MS13 全体の完了条件は以下とする。

- wx 版アプリが起動する
- wx 版メインフレームが成立する
- 基本メニューが表示される
- 基本ボタン列が表示される
- ステータス表示が存在する
- settings load が起動導線へ接続される
- 既存 settings schema を破壊していない
- 将来の worker / file open / process / save 接続点が整理されている
- スプラッシュ、波形、Preview、file dialog 実装を未着手のまま MS14 に送れるよう、境界が明確になっている

## 9. MS13 非完了条件
以下は MS13 で達成していてもよいが、**未達でも MS13 未完了の理由にはしない**。

- recent files の復元
- file dialog 実装
- VMD 保存導線
- 波形表示
- Preview 表示
- 再生機能
- オートスクロール
- 実 worker 稼働
- スプラッシュ
- 完全なテーマ反映
- 現行 Qt 版との見た目 parity

## 10. テスト観点

MS13 時点で確認すべき観点は以下に絞る。

### 10.1 起動
- wx 版が正常起動する
- 起動時クラッシュしない
- 起動入口が一意に定まっている

### 10.2 フレーム構造
- メインフレームが表示される
- レイアウト崩壊がない
- 基本ボタンとメニューが表示される

### 10.3 settings load
- 既存 ini が存在する場合に読み込める
- ini が存在しない場合でも起動できる
- 読込失敗時の最低限のフォールバックがある

### 10.4 将来接続性
- B5 で想定した接続点が過剰抽象化になっていない
- 後続 block で file open / process / save を差し込みやすい

---

## 11. 主なリスク

### 11.1 MS13 の肥大化
最も大きいリスクは、MS13 に parity 回復や file dialog 実装まで入れたくなること。  
これは MS14 の責務なので、MS13 では避ける。

### 11.2 settings 再設計の誘惑
GUI 差し替え時に settings schema を整理したくなる可能性がある。  
しかし今回は既存 schema 維持が前提であり、ここを崩すと原因分離が難しくなる。

### 11.3 worker 本体への先走り
将来の非同期処理を見越して実処理まで入れ始めると、MS13 が重くなる。  
MS13 では受け皿までに留める。

### 11.4 Qt 版との二重保守再開
凍結方針を曖昧にすると、Qt 側にも修正を入れたくなり進行が乱れる。  
MS13 では Qt GUI 凍結を徹底する。

---

## 12. MS14 への引継ぎ前提

MS13 完了後、MS14 では以下を着手対象とする。

- TEXT / WAV 読込導線
- recent files
- settings save の実運用接続
- VMD 保存導線
- warning / busy / status 遷移の parity 回復

つまり MS13 は、MS14 の開始時点で  
**「機能を差し込める wx 主系の土台が揃っている状態」**  
を作るための段である。

---

## 13. 要約

MS13 は、以下の 6 block で進める。

- `MS13-B1`: 移行前提固定と wx 起動入口作成
- `MS13-B2`: メインフレーム最小骨格作成
- `MS13-B3`: メニュー / 基本操作列 / ステータスの最小配置
- `MS13-B4`: settings load 接続と起動時適用
- `MS13-B5`: 将来拡張の接続点整理
- `MS13-B6`: MS13 統合整理と完了確認

この順序により、MS13 は肥大化を避けながら、  
MS14 以降を載せるための wx 主系基盤を成立させられる。

本書を MS13 の block 分解文書として固定し、  
個別 block 実装書は本書の各節を基に切り出すものとする。
