## 0. 文書目的

本書は、`MS13-B2: メインフレーム最小骨格作成` の実装方針を固定するための文書である。

本 block の役割は、`MS13-B1` で成立した wx 主系起動入口の上に、

- 今後の UI 要素を載せるための最小フレーム骨格を作る
- 後続 block である `B3 / B4 / B5` の置き場を明確にする
- まだ未実装の機能領域を、責務を越えない形で placeholder 化する

ことである。

本書は、詳細コード全文を規定するものではない。  
本書は、**Antigravity が MS13-B2 を scope を崩さず実装するための実装計画書**として扱う。

## 2. 本 block で固定する前提

本書は、以下の判断結果を前提とする。

### 2.1 レイアウト粒度
B2 では、ルートパネルの中に以下の 3 分割のみを作る。

- 上部操作域
- 中央主領域
- 下部ステータス域

中央主領域の内部を、さらに左右や上下へ分割するところまでは行わない。

### 2.2 `wx.SplitterWindow` の扱い
- B2 では `wx.SplitterWindow` を導入しない
- 通常の panel / sizer ベースで骨格を作る
- splitter 導入は、後続 block または後続マイルストーンへ送る

### 2.3 placeholder の方針
- 未実装領域は空 panel のまま放置せず、簡単なラベルを置いて役割を見える化する
- 仮ボタンや仮機能導線までは置かない
- 「ここが何の領域か」が分かる最小表現に留める

### 2.4 ステータス域の扱い
- B2 では `wx.StatusBar` を使わない
- 下部ステータス域は通常 panel として作る
- 実際の status 表現の詰めは B3 以降へ送る

### 2.5 settings の扱い
- B2 では settings 適用を扱わない
- ウィンドウサイズ復元、比率復元、初期状態復元は B4 以降へ送る
- B2 は固定レイアウト寄りの最小骨格でよい

## 3. MS13-B2 の目的

MS13-B2 の目的は、以下の 3 点である。

### 3.1 最小フレーム内部構造の成立
`MS13-B1` で表示できるようになった最小 frame の内部に、  
今後の UI が載る土台レイアウトを作る。

### 3.2 後続 block の置き場確保
以下の block が責務に応じて載る領域を先に明確にする。

- `B3`: メニュー / 基本操作列 / ステータスの最小配置
- `B4`: settings load 接続と起動時適用
- `B5`: 将来拡張の接続点整理

### 3.3 構造の先行安定化
機能を急いで入れるのではなく、  
今後大きく作り直さなくて済む最小骨格を先に作る。

## 7. 実装方針

## 7.1 先に「置き場」だけを作る
B2 の本質は、UI の置き場を安定させることである。  
このため、機能本体を先に入れず、以下を先に固定する。

- ルートコンテナ
- 主要領域の分割
- 各領域の役割表示

### 7.2 3 分割に留める
今回は以下の 3 領域だけを作る。

- 上部操作域
- 中央主領域
- 下部ステータス域

中央主領域の内部構造は、この block ではまだ持たせない。

### 7.3 placeholder は最小限に留める
placeholder は、各領域の役割が読めるためのラベルだけでよい。  
仮ボタン、仮テキスト入力、仮ステータス更新などは B3 の責務へ寄るため避ける。

### 7.4 status bar API はまだ固定しない
将来 `wx.StatusBar` を採る可能性を完全には消さない。  
ただし B2 では API を早く固定しすぎず、通常 panel として作る。

### 7.5 settings 由来のレイアウト調整を入れない
settings を見始めると B4 と責務が混ざるため、  
B2 では固定レイアウトの最小骨格に留める。

## 8. 推奨実装順

## Step 1: 既存 main frame / main window の責務確認
まず現行 `main_window.py` を読み、  
以下を大枠で抽出する。

- 上部操作系の責務
- 中央表示系の責務
- 下部状態表示系の責務

この step では、Qt の widget 詳細ではなく、  
領域の責務だけを読み取る。

### この step の成果物
- B2 で置く 3 領域の責務メモ

---

## Step 2: `main_frame.py` にルートパネル追加
`MS13-B1` で成立した最小 frame に対し、  
内部のルートパネルを追加する。

### ここでやること
- frame の内部に root panel を置く
- 今後の sizer 構成の土台を作る

### この step の成果物
- frame 内に UI を積める root panel

---

## Step 3: 3 分割の大枠構造を追加
ルートパネルの中に、以下の 3 領域を通常 panel として作る。

- bottom area

### ここでやること

### ここでまだやらないこと
- 実ボタン

- 中央内部の左右分割
- splitter 導入

### この step の成果物
- 上中下の 3 領域構造

---

## Step 4: placeholder ラベル追加
各領域に、最小のラベルを置いて役割を可視化する。

### 例
- 上部操作域
- 中央主領域
- 下部ステータス域

ラベル文言は厳密固定ではないが、  
役割が明確であることを優先する。

### この step の成果物
- 視覚的に役割が分かる骨格 UI

---

## Step 5: レイアウトの最小整形
極端に見苦しい配置や、今後の追加で崩れやすい構造を避けるため、  
最小限の余白・伸縮・配置方針を整える。

### ここでやること
- sizer での基本整列
- 中央領域が主表示領域として伸びる構造の確認
- 下部ステータス域が過剰に大きくならないよう調整

### この step の成果物
- B3 以降を載せやすい安定骨格

---

## Step 6: B2 範囲の整理と凍結
最後に、B2 で扱う責務と扱わない責務を再確認し、  
B3 へ送るものを明確化する。

### この step の成果物
- B2 完了状態の明確化
- B3 / B4 / B5 へ送る未実装項目一覧

## 13. 想定リスク

### 13.1 B2 に B3 の内容が流入すること
「ついでにボタンも置く」「ついでに status 更新も入れる」などは、  
B3 の責務へ流れやすい。今回は placeholder に留める。

### 13.2 中央領域を先に細分化したくなること
波形 / Preview を見越して左右分割や splitter を入れたくなる可能性がある。  
これは B2 の責務を超えるため避ける。

### 13.3 settings 適用を先に入れたくなること
ウィンドウサイズや見た目調整を先に入れると B4 と混ざる。  
B2 では避ける。

### 13.4 `wx.StatusBar` を早く固定したくなること
status API を先に固めすぎると後続 block の自由度が落ちる。  
B2 では通常 panel に留める。

## 15. 要約

MS13-B2 は、`MS13-B1` で成立した wx 主系起動骨格の上に、

- root panel を追加し
- 上部操作域 / 中央主領域 / 下部ステータス域 の 3 分割を作り
- 各領域を最小 placeholder で可視化する

block である。

この block では、

- splitter は導入しない
- `wx.StatusBar` は使わない
- settings 適用は行わない
- 実機能 UI はまだ入れない

という境界を守る。

本 block の本質は、  
**今後の UI 実装を安全に載せるための最小骨格を、読みやすい形で先に安定化すること**  
である。

本書を `MS13-B2` の実装計画書として固定し、  
Antigravity は本書の範囲内で block 実装を進めるものとする。
