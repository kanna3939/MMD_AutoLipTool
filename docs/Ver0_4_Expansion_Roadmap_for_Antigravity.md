# Ver0.4 拡張ロードマップ概要
## Antigravity 向けマイルストーン土台文書

## 0. 文書目的

本書は、`Ver 0.3.x` 系で初回目標を達成した `MMD_AutoLipTool` を、  
`Ver 0.4.x` 系として再編・拡張していくための**親ロードマップ文書**である。

本書の目的は以下のとおり。

- `Ver 0.4.x` 系の全体像を固定する
- GUI ライブラリ差し替えを最優先課題として明示する
- その後の機能拡張を、手戻りの少ない順で大枠整理する
- Antigravity が個別マイルストーン文書を切る際の基準点とする
- 実装対象と非対象を先に明文化し、作業中の論点拡散を防ぐ

本書は詳細実装書ではない。  
本書は、**Ver0.4 全体の進め方を固定するための上位方針文書**として扱う。

## 1. Ver0.4 の位置づけ

### 1.1 基準時点

- ベースライン: `Ver 0.3.x`
- 前提:
  - 既存ツールは公開可能な初回到達点に到達済み
  - `Ver 0.3.x` は Qt Widgets / PySide6 ベースで構築されている
  - `Ver 0.4.x` では GUI 基盤の見直しと、その後の機能拡張を行う
  - `Ver 0.4.x` の実装作業は Antigravity を主作業者として進める

### 1.2 Ver0.4 の主テーマ

`Ver 0.4.x` の主テーマは、単なる機能追加ではない。  
最初の主題は次の 2 点である。

1. **GUI 基盤を Qt / PySide6 から wxPython へ差し替える**
2. **差し替え後の安定基盤の上で、入力拡張・品質改善・UX 改善を進める**

このため、`Ver 0.4.x` は次の二層構造で捉える。

- 前半:
  - GUI 基盤移行
  - 既存機能 parity 回復
  - 再生系 UX 再構築
- 後半:
  - 入力拡張
  - 生成品質改善
  - 将来的な多言語対応

## 2. Ver0.4 で解決したい課題

### 2.1 GUI ライセンス運用上の扱いづらさ

現行の Qt / PySide6 ベース実装は機能的には成立しているが、  
公開・配布・将来運用の観点では、GUI ライブラリ周辺の扱いがやや重い。

そのため Ver0.4 では、GUI 基盤を `wxPython` に移行し、  
今後の配布・保守・拡張の土台をより扱いやすい形へ寄せる。

### 2.2 GUI 基盤変更前提での機能追加の非効率

現行 GUI に新機能を積み増すと、後で wxPython へ移植する際に  
同じ機能を 2 回作る形になりやすい。

そのため、GUI に強く結びつく新機能は、  
**wxPython 移植後に実装する方針**を採る。

### 2.3 公開後に必要になる実用拡張

公開後の利用を想定すると、以下の方向性が有用である。

- TEXT なし利用への拡張
- mp3 入力対応
- 再生中の視認性向上
- 連続母音時の品質改善
- 将来的な英語対応

Ver0.4 は、これらを無秩序に追加するのではなく、  
**GUI 基盤移行後の安定順に実装する**ことを重視する。

## 4. Ver0.4 の設計原則

### 4.1 core と GUI の責務分離を維持する

既存 repo では、比較的 `core` / `vmd_writer` と GUI 層が分離されている。  
Ver0.4 でもこの方針は維持する。

- `core`
  - 音声解析
  - テキスト処理
  - タイミング生成
- `vmd_writer`
  - VMD 出力
  - 形状生成
  - 最終正規化
- `gui`
  - 入出力導線
  - 設定
  - 状態遷移
  - 可視化
  - 再生時の同期

### 4.2 GUI 基盤差し替えと機能追加を同時に広げすぎない

Ver0.4 の最大リスクは、  
**GUI 差し替えと新機能追加が同時進行で広がり、原因分離ができなくなること**である。

そのため、まずは wxPython 上で既存機能を安定させる。  
その後に、再生 UX と入力拡張を積む。

### 4.3 既存資産を活かす

以下はできるだけ既存資産を活かす。

- `core` の音声解析
- `core` の text to hiragana / vowel 系
- `pipeline` の timing plan
- `vmd_writer` の observation ベース出力
- 既存 settings schema の考え方

### 4.4 段階的に完成度を上げる

Ver0.4 は、一気に完成形へ到達させるのではなく、  
以下の順に段階的に上げる。

1. 起動する
2. 既存機能が使える
3. 再生体験が改善する
4. 入力手段が増える
5. 出力品質が上がる
6. 対応言語が広がる

## 5. 実装の大枠順序

Ver0.4 の大枠実装順は、次の順で固定する。

#### 目的

#### 主内容
- アプリ起動骨格の置換
- メインフレーム構築
- メニュー / ボタン / ステータス導線置換
- 設定ロード / 保存接続
- ダイアログ導線置換

#### 完了条件
- wxPython 版アプリが起動する
- 基本操作導線が存在する
- 設定永続化が機能する

---

### Phase 2: 既存機能 parity 回復

#### 目的
現行版で使える最低限の日常機能を wx 版に揃える。

#### 主内容

- recent files
- VMD 保存
- 保存先フォルダ記憶
- 処理実行

- エラー警告

---

### Phase 3: 波形表示・Preview 表示の復元

- Preview 表示移植
- 共有 viewport / 位置同期の復元
- ズーム / pan の基礎復元

#### 完了条件
- 波形と Preview が表示できる
- 解析結果の視覚確認が成立する

---

### Phase 4: 再生系 UX 再構築

#### 目的
wx 基盤に合わせて、再生まわりの体験を立て直す。

#### 主内容
- 再生位置追従
- オートスクロール
- 表示更新頻度の整理
- 再生中描画の滑らか化
- GUI スレッドと非 GUI スレッドの役割整理

#### 完了条件
- 再生位置に応じて表示が自然に追従する
- 体感上の引っかかりが減る
- 再生中の視認性が改善する

### Phase 5: 入力拡張 1

## 9. 主なリスク

### 9.1 GUI 移植の長期化
wxPython 移植は局所修正ではなく、GUI 層の広範囲差し替えになる。  
そのため、最初のフェーズが長期化しやすい。

### 9.2 parity 不足
wx 版が起動しても、現行版で当たり前にできる操作が欠ける可能性がある。  
そのため、早期に parity 観点を固定する必要がある。

### 9.3 再生系の挙動差
再生位置更新、描画頻度、スクロール追従は、GUI 基盤差し替えで体感差が出やすい。  
Ver0.4 では、この領域を独立したフェーズとして扱う。

### 9.4 品質改善の論点拡散
同音母音谷改善や発音単位スロープ設定は、  
writer / preview / pipeline 契約にまたがる。  
GUI 移植と同時に広げると整理不能になりやすい。

### 9.5 英語対応の過大化
英語対応は ASR 追加だけでは済まず、口形マッピング設計が本体になる。  
そのため、後段に回す。
