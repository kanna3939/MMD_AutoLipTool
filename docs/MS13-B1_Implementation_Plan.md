## 2. 本 block で固定する前提

本書は、以下の判断結果を前提とする。

### 2.1 `main.py` の扱い
- `src/main.py` を wx 主系へ直接差し替える
- `main.py` は Qt / wx の振り分け器にはしない
- Ver0.4 系では `main.py` を wx 起動入口の正本とする

### 2.2 wx GUI パッケージ配置
- wx 実装は `src/gui_wx/` に新設する
- 既存 `src/gui/` は直ちに削除しなくてもよいが、MS13 以降の改修対象から外す
- `src/gui/` は Qt 系の凍結領域、`src/gui_wx/` は新主系領域として扱う

### 2.3 Windows 固有起動要素
- アイコン設定
- AppUserModelID
- スプラッシュ
- 起動演出全般

これらは `MS13-B1` では非対象とする。

## 7. 実装方針

## 7.1 起動責務を最小化する
`src/main.py` は、Ver0.4 以降の wx 主系入口として整理する。  
ここでは責務を広げすぎず、以下に絞る。

- wx アプリ生成
- 最小フレーム生成
- フレーム表示
- メインループ開始

settings 適用や画面構築の本体は、B2/B4 に送る。

### 7.2 `gui_wx` を新主系として独立させる
`src/gui_wx/` は、Qt の置換先ではなく、Ver0.4 系の新しい GUI 正本として扱う。  
このため、以下を意識する。

- Qt 系ファイルを流用しすぎない
- package 構成は今後の block 追加に耐えるようにする
- `gui_wx` 配下に責務を素直に積める形にする

### 7.3 Qt 側は凍結扱いにする
Qt 側のコードは参照してよいが、  
本 block で Qt 側にも追加修正を広げない。

必要最小限の import 切り替えや起動経路整理は許容するが、  
Qt GUI の改善や parity 補修は行わない。

### 7.4 将来接続を意識しつつ、過剰抽象化しない
B1 は土台 block だが、ここで抽象層を厚く作りすぎない。  
「将来使える」ことと「今わかりやすい」ことの両立を優先する。

## 8. 推奨実装順

## Step 1: 既存 `main.py` の責務確認
まず現行 `src/main.py` を読み、以下を分離する。

- Qt 固有責務
- GUI 共通責務
- 今の B1 に不要な責務

この確認により、B1 で持つべき責務を最小化する。

### この step の成果物
- B1 で残す責務一覧
- B1 では持たない責務一覧

---

## Step 2: `src/gui_wx/` package 新設
新主系 GUI の置き場として、`src/gui_wx/` を作る。

### 最小構成の例
- `src/gui_wx/__init__.py`
- `src/gui_wx/app.py`
- `src/gui_wx/main_frame.py`

ファイル名は厳密固定ではないが、  
責務は「app 入口」と「frame 最小骨格」に分けることを推奨する。

### この step の成果物
- wx 主系 package
- import 可能な最小 module 構造

---

## Step 3: wx アプリ起動導線作成
`wx.App` を使い、アプリが起動できる最小導線を作る。

### ここでやること
- app クラスまたは app 生成関数の定義
- 最小 main frame 生成
- show 呼び出し
- main loop 呼び出し

### ここでまだやらないこと
- 複雑な初期化
- settings 適用
- メニュー構築
- ボタン配置完成

### この step の成果物
- `python src/main.py` 相当で wx アプリ起動

---

## Step 4: `src/main.py` を wx 主系へ切り替え
既存 `main.py` を、Qt 主系ではなく wx 主系入口として差し替える。

### ここでの注意
- 一時的な Qt/wx 分岐器にはしない
- wx が Ver0.4 の正本であることを明確にする
- 旧 Qt 起動コードは安易に混在させない

### この step の成果物
- `main.py` が wx 起動入口になる

---

## Step 5: 最小 frame 骨格の成立確認
B2 の前提となる、空でもよい最小 frame を成立させる。

### 最低限含めるもの
- タイトル
- フレーム生成
- 閉じることができる
- 将来 panel を置ける余白

### この step の成果物
- 今後のレイアウト実装が載る器

---

## Step 6: B1 範囲の整理と凍結
最後に、B1 で扱う責務と扱わない責務を整理し、  
B2 に送る部分を明確にする。

### この step の成果物
- B1 完了状態の明確化
- B2 へ渡す未実装項目一覧

## 13. 想定リスク

### 13.1 過渡期構造の混入
`main.py` に Qt/wx 両対応ロジックを残すと、B1 の責務が濁る。  
今回は wx 主系へ直接切り替える前提を守る。

### 13.2 `gui_wx` ではなく既存 `gui` を直接いじりたくなること
今回の前提では `gui_wx` を新主系とする。  
途中で既存 `gui/` を直接上書きすると、凍結境界が曖昧になる。

### 13.3 B1 に B2/B4 の内容が流入すること
メニュー、ボタン、settings 適用を入れたくなっても、  
B1 では起動入口と最小 frame に留める。

### 13.4 Windows 固有演出の流入
アイコン、AppUserModelID、スプラッシュは scope 外。  
ここで入れると block が膨らむ。

## 15. 要約

MS13-B1 は、Ver0.4 系の起点として、

- `src/main.py` を wx 主系起動入口へ差し替え
- `src/gui_wx/` を新主系 GUI package として新設

する block である。

**wx 主系へ移行したことが構造上明確にわかる最小骨格の成立**を最優先とする。

本書を `MS13-B1` の実装計画書として固定し、  
Antigravity は本書の範囲内で block 実装を進めるものとする。
