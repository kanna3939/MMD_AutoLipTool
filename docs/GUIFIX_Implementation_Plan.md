# GUIFIX 実施報告書 (Implementation Plan Summary)

本ドキュメントは、MMD AutoLip Tool において実施された一連のGUIレイアウトおよび視認性向上のための修正（GUIFIX01〜05）についてまとめたものです。

## 実施期間
2026-03-23

## 修正内容一覧

### GUIFIX01: 左パネルのレイアウト破綻防止
- **課題**: ウィンドウサイズが小さい時や長文入力時に、左パネルのウィジェット（テキスト枠）が重なって表示される。
- **対策**: `LeftInfoPanel` 全体を `QScrollArea`（スクロール領域）で包み、コンテンツが溢れた場合にスクロール可能に修正。

### GUIFIX02: 入力テキスト確認枠の最適化
- **課題**: テキスト確認枠（3カ所）が縦に長く、初期状態で画面に収まりきらない。
- **対策**: 各テキスト枠の初期表示行数を約5行分（高さ80px）に制限し、垂直方向のコンパクト化を実現。

### GUIFIX03: ウィンドウサイズ拡張とフォントの統一
- **課題**: 全体的に表示が窮屈で、WAV情報などの一部が見切れる。
- **対策**: 
    - 初期ウィンドウサイズを `1270x714` から `1280x800`（後に `1280x740` へ調整）に拡大。
    - 左パネル内のフォントサイズを 8pt（後に 9pt へ調整）に一括調整。
    - WAV情報ラベルに最小高さを設定し、見切れを解消。

### GUIFIX04: フォント未反映問題の解消
- **課題**: `setFont()` による指定がグローバルスタイルシート（QSS）によって無効化されていた。
- **対策**: ウィジェット個別の `setStyleSheet("font-size: 9pt;")` を用いて、グローバル設定を優先的にオーバーライドするように修正。

### GUIFIX05: メニュー・ボタンのOSネイティブフォント採用
- **課題**: トップメニューや操作ボタンの文字が、本文用設定（11pt等）に引きずられて大きく表示される。
- **対策**: 
    - メニューバー、メニュー項目、操作ボタンのスタイルシートからフォント指定を除外し、Windows標準のシステムフォントサイズに自動追従するように変更。
    - 操作ボタンのアイコンサイズを `24x24` ピクセル（中サイズ）として明示し、視認性と操作性を向上。

## 適用後の主要な仕様
- **初期ウィンドウサイズ**: 1280 x 740 px
- **パネル内フォント**: 9pt (スタイルシート強制適用)
- **メニュー・ボタンフォント**: OS標準設定 (Segoe UI / 游ゴシック UI 等)
- **ボタンアイコン**: 24 x 24 px
- **メニュー言語**: 日本語統一（ファイル、処理、ビュー、ヘルプ）

## GUIFIX06_Fase1_Result

### 1. フェーズ1実施範囲
- 確認対象ファイル:
  - `.\src\gui\waveform_view.py`
  - `.\src\gui\preview_area.py`

  - `.\src\gui\main_window.py`
- 実施したステップ: フェーズ1のステップ1-1〜1-4

### 2. `WaveformView` 側の現行基準整理
- **plot area 決定箇所**: 明示的な固定値設定はなく、Matplotlib の `FigureCanvasQTAgg` および `Figure.tight_layout()` に完全に依存している。描画データの有無・桁数によって Y軸ラベルの幅が変わり、結果的に plot area 左端座標も動的に変動し得る構造。
- **横軸可視範囲決定箇所**: `_resolved_visible_range_sec()` で `_viewport_start_sec` と `_viewport_end_sec` を計算し、`self._axes.set_xlim(...)` で Matplotlib 側に設定している。
- **Matplotlib レイアウト依存箇所**: Y軸のラベル桁数、各種 padding、グリッドマージンなどの計算は Matplotlib が内包しており、PySide6 側からは（現状のコードでは）最終的な描画矩形の `QRectF` として容易に取り出せる口が用意されていない。
- **リサイズ影響箇所**: ウィンドウのリサイズに応じて `FigureCanvas` がリサイズされ、それに追従して内部の axes が変形する。

### 3. `PreviewArea` 側の現行基準整理
- **`timeline_rect` 決定箇所**: `paintEvent` 内でウィジェットの `self.rect()` を取り、そこから `_OUTER_MARGIN` (8px), `_LABEL_WIDTH` (18px), `_LABEL_TIMELINE_GAP` (6px) を引いた領域を手計算で作っている。
- **左ラベル列幅**: `_LABEL_WIDTH` (18px) として固定。
- **グリッド開始X**: `timeline_rect.left()` （キャンバス左端 + 8 + 18 + 6 = 32px の位置）。ここに Waveform 側の動的マージン（Y軸の幅分など）は一切考慮されていない。
- **有効描画幅**: `timeline_rect.width()`。これも全体の幅から固定マージンを引いた値であり、Waveform 側の内部幅とは異なる。
- **描画対象ごとの基準矩形**:
  - フレーム軸ティック、行のグリッド、セグメント矩形、再生位置カーソルのいずれも、この固定手計算された `timeline_rect` の左端と幅を基準 (`_seconds_to_x`) に描画されている。

### 4. 横方向変換の整理
- **`WaveformView` 側の入力値**: `_viewport_start_sec`, `_viewport_end_sec`, `_duration_sec`, および Matplotlib が管理する `axes` の表示領域情報。
- **`PreviewArea` 側の入力値**: `_viewport_start_sec`, `_viewport_end_sec`, `_timeline_duration_sec`, および `timeline_rect` (幅・左端)。
- **共有されている値**: `ViewSync` 経由で `shared_viewport_sec` (`_viewport_start_sec`, `_viewport_end_sec`) と `shared_position_sec` (再生カーソル位置) は正確に共有されている。
- **共有されていない値**: 画面上のどこからどこまで（ピクセル座標）にその秒範囲をマッピングするか、という **「物理描画領域（x座標と幅）」の情報** が共有されていない。
- **`ViewSync` の責務範囲**: 現在の `ViewSync` は「秒・時間の状態共有」のみを行っており、ピクセルや描画矩形などの「ビュー固有のレイアウト寸法」は管理していない。

### 5. フェーズ1時点の責務境界整理
- **今回の問題をどの責務層の問題として扱うべきか**: データ層や論理同期層（`ViewSync`）のエラーではなく、「表示コンポーネント間のレイアウト連携層（ビュー・プレゼンテーション層）」の不足として扱うべき。
- **なぜ単なる見た目調整問題ではないか**: Preview側の左マージン固定値 (32) を微調整しても、Waveform側の Y軸ラベル幅の変化（秒数や時間の長さで動的に変わる）やOS/フォントごとの文字幅の違いによって容易に再発するため。根本的な描画開始基準 x 座標を動的に一致させる必要がある。
- **なぜ `ViewSync` 単体の問題ではないか**: `ViewSync` はロジック上の「時間」を扱うべきであり、「画面の右から何ピクセル目か」という GUI 依存の色合いが強い情報を乗せるべきかについては設計上の検討を要する（MainWindow などのレイアウト管理側で直接渡す方が素直な可能性がある）。

### 6. 修正対象外の明示
- **今回フェーズ1で対象外とした項目**:
  - 修正コードの実装
  - GUI全体のレイアウト再調整
- **後続フェーズで扱う項目**:
  - Waveform 側からの実 plot area の抽出実装（PySide と matplotlib の座標変換）
  - 抽出した矩形情報を Preview 側へ流し込み、独自の固定余白推測計算 (`_LABEL_WIDTH`等) を捨てて座標を一致させる修正
- **この段階で未確定のまま残す項目**:
  - 矩形情報を「`ViewSync` に追加する」か、「`MainWindow` / `RightDisplayContainer` のレイアウト更新時に直接渡す」か、という具体的なルーティング設計。

### 7. フェーズ2へ引き継ぐ要点
- **正基準固定のために必要な材料**: `WaveformView` の `_axes.get_window_extent()` などを利用し、PySide6 側のウィジェット座標系における plot area の矩形（特に `left` と `width`）を取得する関数が必要。
- **一致対象定義に必要な材料**: この左端位置 `x` を `PreviewArea` 側の `timeline_rect.left()` として直接適用し、幅も同じように乗算できる経路。
- **現時点で有力な論点**: matplotlib の再描画（リサイズやデータの初回流し込みに伴う ticks の再構築）によって plot area のサイズが変わるタイミングを正しくキャッチし、`PreviewArea` に更新をかけるイベントフロー（`resizeEvent` や再描画シグナルなど）をどう構築するかが鍵となる。

- **参照した既存情報の範囲**:

  - `WaveformView` 側の Matplotlib 依存による動的描画領域の構造

  - `ViewSync` の現在の責務（論理時間のみの同期）

### 2. 横軸正基準の固定
- **`WaveformView` を正基準とする判断**: `WaveformView` 側を横軸のマッピング正基準として固定する。
- **その理由**: `WaveformView` の Y軸ラベル（振幅値や文字列）は可変であり、Matplotlib が最適なレイアウトを動的に計算・維持しているため、これを固定値で強制すると波形自体の描画効率や見栄えが著しく損なわれる。
- **`PreviewArea` を追従側とする整理**: `PreviewArea` は手描きのカスタムウィジェットであり、動的な余白を外部から挿入・調整しやすい構造になっている。よって、`PreviewArea` 側を追従させる構成が最も自然である。
- **この方針で固定してよい理由**: GUIの実装難度および既存の柔軟性を活かす観点で、Matplotlibの内部処理に手を入れるより、独自描画のQPainter側を適応させるほうが安全で確実度が高いため。

### 3. 一致対象の固定
- **Waveform の実 plot area 左端**: `PreviewArea` のタイムライン（グリッド開始位置）の `x` コンポーネントをここに合わせる。
- **Waveform の実 plot area 幅**: `PreviewArea` の描画領域の幅をここに合わせる。
- **同一時間が同一 x に投影されること**: 同一の「秒 (seconds)」が、両ビューで完全に同一のピクセル `x` にマッピングされる状態。
- **この3点を一致対象とする理由**: 実際の「見た目のズレ」は、始点の不一致とスケール（幅）の不一致が複合して起きているため。
- **左端だけでは不十分な理由**: ウィジェット全体の幅が同じであっても、右端のマージン計算（Matplotlib側のpadding等）が異なると、右へ行くほど時間がズレてしまう。
- **幅だけでは不十分な理由**: 描画領域の幅が同じでも、左マージンの開始位置がずれていれば、全体が平行移動した状態でずれ続けるため。

### 4. 一致判定基準の固定
- **内部描画矩形を基準とすること**: `WaveformView` 全体のウィジェット矩形 (`QRect`) ではなく、「波形が実際に描画される領域（Axes の内側）」を基準とする。
- **なぜその基準が必要か**: 波形グラフには外側に軸ラベル、タイトル、余白などの占有領域が存在するため。
- **リサイズやレイアウト変化に対して有効な理由**: ウィンドウのサイズ変更によって利用可能な総ピクセル数が変わった場合でも、「純粋なプロット領域の比率」として再計算・再同期させれば常に同一時間同一表示を保証できる。
- **外枠基準で起こりうる問題**: `PreviewArea` 側と `WaveformView` 側の「枠外（ラベル等）に必要な余白」の量が異なるため、外枠を一致させても内部のグラフ領域は決して合致しない。

### 5. `PreviewArea` における一致対象範囲
- **左ラベル列を一致対象から外す整理**: `PreviewArea` の左ラベル（母音の "あ" など）は、Waveform の Y軸ラベルとは全く別の情報・意味合いを持つため、この幅を Waveform に合わせる必要はない。
- **グリッド開始位置以降の時間軸領域を一致対象とする整理**: ラベル列より右側の、「実際にイベント矩形やグリッド線が描画される領域」のみを一致対象として切り出す。
- **その利点**: 左ラベル列のデザインや幅を Waveform のY軸幅とは無関係に独立して維持でき、見た目の破綻を防げる。

### 2. 基準矩形の再取得整理
- **どの契機で Waveform 側の最新基準矩形を取り直すようにしたか**:
  - Matplotlib canvas の `draw_event` および `resize_event` を介した再取得。
  - `xlim_changed` コールバックによる軸範囲変化時の再取得。
  - これらのイベントは `MainWindow` がフックし、`_request_waveform_bounds_sync()` を通じて遅延同期（QTimerによる一括処理）を行っている。
- **初期表示時との違い**: 初期表示時はレイアウト確定前の不確定な値を避けるため、スプリッター比率適用後に1回明示的に要求するが、リサイズ・ズーム・パン時は Matplotlib の内部再計算が完了したタイミング（`draw_event`）を待って最新の物理ピクセル座標を取得する。

### 3. 再通知・再反映の整理
- **どの契機で Preview 側へ再通知するようにしたか**: `WaveformView` の描画サイクルが一段落した直後。
- **値未確定時の扱い**: `WaveformView.get_plot_area_rect()` が `None` を返す場合や、描画データがない場合は、`PreviewArea` 側で安全に既存の固定マージン計算へフォールバックされる。
- **再描画との関係をどう整理したか**: `PreviewArea.set_waveform_plot_area_rect()` が呼ばれると即座に `self.update()` が走り、次の描画タイミングで最新矩形が反映される。
- **片更新を防ぐために何をしたか**: `_request_waveform_bounds_sync` による遅延実行（`singleShot(0)`）を用いることで、一連のレイアウト変化（リサイズ等）が落ち着いた後に同期が行われるようにし、描画のバタつきを抑えた。

### 4. Zoom / Pan / レイアウト変化への対応整理
- **Zoom 時に何を維持したか**: `shared_viewport_sec` の更新後に Waveform の `draw_idle` が走り、その後の `draw_event` で再整合される流れ。
- **Pan 時に何を維持したか**: `_on_pan_requested` やスクロールバー操作時に、`_request_waveform_bounds_sync()` を明示的に呼び出すパスを追加し、リアルタイムな追従性を高めた。
- **splitter や横幅変化時に何を維持したか**: `splitterMoved` 信号から直接同期を要求し、スプリッター操作中も座標系が追従するようにした。
